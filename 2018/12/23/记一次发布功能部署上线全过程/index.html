记一次发布功能部署上线全过程

发表于 2018-12-23 | 分类于 git
git在企业开发的基本流程

基于master分支checkout 一个新分支，比如feature/xxx (git checkout -b feature/xxx)

这样就在新建的本地分支上进行开发

若是和其他同事协同开发，需推送功能分支到远端，(git push —set-upstream origin feature/xxx)

其他同事则检出前面的新分支(git pull && git checkout feature/xxx )
完成开发，合并分支时如果遇到冲突，前端同学合并前端代码，后端同学合并后端代码，两人最好聚在一起分析冲突，避免合并错误

提交测试，合并到develop (git checkout develop && git merge —no-ff feature/xxx)

在pull上传的代码的时候

注意：git push origin develop:线上开发分支 中 ： 冒号前后一定不要有空格，如果有空格，会直接push到空格前后的两个分支上！：冒号前后一定不要有空格！！！

注意：walle部署的时候测试环境部署选择的分支是develop_jenkins，正式生产环境选择的分支是release_jenkins

在jenkins生成测试分支(develop_jenkins)，然后在walle的预发布环境中选择测试分支，发布代码到测试机

jenkins构建与walle部署与下面提交正式环境类似，不多做赘述

测试完成，合并到master，此时需要使用wpsgit的merge request方式合并，指定审核人，由审核人合并

分支合并完成之后在jenkins中进行项目构建

在正式上线环境中发布部署

随后部署上线

上线成功

注意 ： 完整的过程需要先在develop中进行合并测试，测试通过才可以合并到master分支上。
